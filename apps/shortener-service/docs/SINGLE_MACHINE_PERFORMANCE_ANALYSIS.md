# 单机性能上限分析

## 概述

本文档分析短链服务在单机环境下的理论性能上限，并提供达到 50 万 QPS 的扩展方案。

## 测试环境假设

### 硬件配置（标准单机）

- **CPU**: 8 核
- **内存**: 16GB
- **网络**: 千兆网卡 (1Gbps)
- **磁盘**: SSD

### 软件配置

- **Go 服务**: 1 实例
- **MySQL**: 1 实例（本地或同机房）
- **Redis**: 1 实例（Standalone 模式）

## 性能瓶颈分析

### 1. 重定向请求（GET /:code）- 主要场景

重定向是最高频的操作，性能分层如下：

#### L1 缓存命中（Ristretto 内存缓存）

- **延迟**: ~0.1ms（内存访问）
- **吞吐**: 200K-300K QPS
- **瓶颈**: CPU 和 Go runtime
- **命中率**: 70-80%

#### L2 缓存命中（Redis）

- **延迟**: ~1-2ms（网络 + Redis 处理）
- **吞吐**: 80K-120K QPS
- **瓶颈**: Redis 单线程处理 + 网络 I/O
- **连接池配置**: PoolSize=20-30
- **命中率**: 15-20%

#### 数据库查询（MySQL）

- **延迟**: ~5-10ms（有索引）
- **吞吐**: 5K-10K QPS
- **瓶颈**: MySQL 连接池 + 磁盘 I/O
- **命中率**: 5-10%

#### 综合性能计算

**缓存命中率分布：**
- L1 命中率: 75%
- L2 命中率: 20%
- DB 查询: 5%

**加权平均 QPS：**
```
总 QPS = (L1命中率 × L1_QPS) + (L2命中率 × L2_QPS) + (DB命中率 × DB_QPS)
       = (0.75 × 250K) + (0.20 × 100K) + (0.05 × 8K)
       = 187.5K + 20K + 0.4K
       ≈ 208K QPS
```

**实际单机上限（考虑其他开销）：**
- **重定向请求: 150K-180K QPS**

### 2. 创建短链（POST /api/v1/CreateShortLink）

#### 性能分析

- **MySQL INSERT**: ~2-5ms
- **Redis SET**: ~1ms
- **ID 生成**: ~0.1ms
- **URL 验证**: ~0.1ms
- **总延迟**: ~3-6ms

#### 吞吐上限

- **MySQL 写入瓶颈**: 8K-12K QPS
- **Redis 写入**: 80K+ QPS（不是瓶颈）

**实际单机上限：**
- **创建请求: 8K-10K QPS**

### 3. 混合负载（80% 读 + 20% 写）

```
混合 QPS = (0.8 × 重定向QPS) + (0.2 × 创建QPS)
         = (0.8 × 150K) + (0.2 × 10K)
         = 120K + 2K
         = 122K QPS
```

**实际单机上限（混合负载）：**
- **混合请求: 100K-120K QPS**

## 关键瓶颈点

### 1. MySQL 写入性能

- **单机 MySQL 写入**: 8K-12K QPS
- **瓶颈原因**: 
  - 磁盘 I/O（即使是 SSD）
  - 事务日志（binlog, redo log）
  - 索引维护开销

**优化方向：**
- 批量插入（Batch Insert）
- 异步写入队列
- 分库分表
- 使用 NVMe SSD

### 2. Redis 单线程限制

- **Redis 单实例**: 80K-120K QPS
- **瓶颈原因**:
  - 单线程处理模型
  - 网络 I/O
  - 内存访问延迟

**优化方向：**
- Redis Cluster（水平扩展）
- 读写分离（主从复制）
- Redis 6.0+ 多线程 I/O

### 3. 网络带宽

- **千兆网卡**: ~125MB/s
- **假设平均响应 1KB**: 125K QPS
- **实际可用**: ~100K QPS（考虑协议开销）

**优化方向：**
- 万兆网卡（10Gbps）
- 响应压缩
- HTTP/2 或 gRPC

### 4. Go 服务 CPU

- **8 核 CPU**: 每核处理 ~25K QPS
- **理论上限**: 200K QPS
- **实际上限**: 150K-180K QPS（考虑 GC 和上下文切换）

**优化方向：**
- 增加 CPU 核心数
- 优化 GC 参数
- 减少内存分配

## 单机性能总结

| 场景 | 单机 QPS | P99 延迟 | 备注 |
|------|---------|---------|------|
| **纯重定向（高缓存命中）** | 150K-180K | < 5ms | L1+L2 命中率 95% |
| **纯创建（写入密集）** | 8K-10K | < 50ms | MySQL 写入瓶颈 |
| **混合负载（80读20写）** | 100K-120K | < 10ms | 实际生产场景 |
| **极限优化单机** | 280K-300K | < 5ms | 需要顶配硬件 |

## 达到 50 万 QPS 的方案

要达到 **50 万 QPS**，单机是**不可能**的。需要水平扩展架构。

### 方案 1: 水平扩展（推荐）

#### 架构设计

```
                    Envoy/Nginx 负载均衡
                            ↓
        ┌───────────────────┼───────────────────┐
        ↓                   ↓                   ↓
   短链服务 × 5 实例（每个 100K QPS）
        ↓                   ↓                   ↓
        └───────────────────┼───────────────────┘
                            ↓
              Redis Cluster × 6 节点
                    (3 主 3 从)
                            ↓
              MySQL 主从 + 读写分离
                    (1 主 2 从)
```

#### 配置建议

**1. 短链服务（5 实例）：**

```bash
# 每实例配置
CPU: 8 核
内存: 16GB
网络: 万兆网卡

# 环境变量
REDIS_POOL_SIZE=30
REDIS_MIN_IDLE_CONNS=9
REDIS_CONN_MAX_LIFETIME=45m
REDIS_READ_TIMEOUT=2s
REDIS_WRITE_TIMEOUT=2s

# MySQL 连接池
MYSQL_MAX_OPEN_CONNS=50
MYSQL_MAX_IDLE_CONNS=10
```

**2. Redis Cluster（6 节点）：**

```bash
# 每节点配置
CPU: 4 核
内存: 8GB
网络: 万兆网卡

# Redis 配置
maxmemory: 6GB
maxmemory-policy: allkeys-lru
tcp-backlog: 511
timeout: 0
tcp-keepalive: 300
```

**3. MySQL（1 主 + 2 从）：**

```bash
# 主库（写入）
CPU: 16 核
内存: 32GB
磁盘: NVMe SSD RAID 10

# MySQL 配置
max_connections: 500
innodb_buffer_pool_size: 24GB
innodb_log_file_size: 2GB
innodb_flush_log_at_trx_commit: 2

# 从库（读取）
CPU: 8 核
内存: 16GB
磁盘: SSD

# MySQL 配置
max_connections: 300
innodb_buffer_pool_size: 12GB
```

#### 预期性能

- **重定向**: 500K QPS
- **创建**: 50K QPS
- **P99 延迟**: < 5ms
- **可用性**: 99.9%
- **缓存命中率**: > 95%

#### 成本估算

**云服务器（按月）：**
- 短链服务: 5 × 2000 元 = 10,000 元
- Redis Cluster: 6 × 1500 元 = 9,000 元
- MySQL: 3 × 5000 元 = 15,000 元
- 负载均衡: 2 × 1000 元 = 2,000 元
- 网络流量: ~10,000 元
- **总计: ~46,000 元/月**

**自建机房（一次性 + 运维）：**
- 服务器硬件: 16 台 × 2 万 = 32 万元
- 网络设备: 5 万元
- 机柜租赁: 2 万元/年
- 运维人力: 1 万元/月
- **总计: ~37 万元（一次性）+ 1 万元/月**

### 方案 2: 极致优化单机（理论上限）

如果必须单机，可以达到 **30-35 万 QPS**（极限）。

#### 硬件配置

- **CPU**: 64 核（AMD EPYC 或 Intel Xeon）
- **内存**: 128GB DDR4 ECC
- **网络**: 万兆网卡（10Gbps）
- **磁盘**: NVMe SSD RAID 0（4TB）

#### 软件优化

1. **Redis Cluster 本地部署**（6 节点）
2. **MySQL 只写主键索引**，异步写入详细数据
3. **L1 缓存扩大到 8GB**
4. **Go 服务优化**：
   - GOMAXPROCS=64
   - GOGC=200（减少 GC 频率）
5. **内核参数优化**：
   ```bash
   net.core.somaxconn=65535
   net.ipv4.tcp_max_syn_backlog=65535
   net.ipv4.ip_local_port_range=1024 65535
   ```

#### 预期性能

- **L1 命中率提升到 90%**: 90% × 300K = 270K QPS
- **L2 命中率**: 8% × 120K = 9.6K QPS
- **DB 查询**: 2% × 10K = 0.2K QPS
- **总计**: ~280K QPS

#### 缺点

- **成本极高**: 单机硬件 > 10 万元
- **单点故障风险**: 无冗余
- **扩展性差**: 无法继续水平扩展
- **不推荐用于生产环境**

## 推荐方案总结

### 目标: 50 万 QPS

**推荐架构**: 方案 1（水平扩展）

**关键配置：**

| 组件 | 实例数 | 单实例 QPS | 总 QPS |
|------|--------|-----------|--------|
| 短链服务 | 5 | 100K | 500K |
| Redis Cluster | 6 (3主3从) | 80K | 480K |
| MySQL | 1主2从 | 写10K/读50K | 写10K/读100K |

**部署建议：**

1. **使用 Kubernetes 编排**
2. **配置 HPA（水平自动扩展）**
3. **监控告警**：
   - CPU > 70% 自动扩容
   - 内存 > 80% 告警
   - P99 延迟 > 10ms 告警
4. **灰度发布**：
   - 金丝雀发布（10% → 50% → 100%）
   - 自动回滚机制

## 性能测试计划

### 测试场景

1. **基准测试**：
   - 纯重定向: 目标 150K QPS
   - 纯创建: 目标 10K QPS
   - 混合负载: 目标 100K QPS

2. **压力测试**：
   - 持续高负载: 100K QPS × 10 分钟
   - 峰值冲击: 0 → 200K QPS × 1 分钟

3. **稳定性测试**：
   - 长时间运行: 50K QPS × 24 小时
   - 故障恢复: Redis 宕机恢复测试
