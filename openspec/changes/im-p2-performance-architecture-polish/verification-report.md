## Capacity Model & Bottleneck Analysis (Task 1.3, non-loadtest edition)

### Scope and Assumptions

This section provides a **non-loadtest** capacity model based on:

1. gateway runtime configuration and critical path code,
2. completed regression/integration evidence,
3. observed behavior constraints introduced by P2 optimizations.

It is intended as a planning model for architecture and risk prioritization, not a replacement for staging pressure baselines.

### Model Inputs (from code paths)

- Connection-level ingress guard: `GatewayConfig.MaxMessagesPerSecond = 100`.
- ACK timeout default: `GatewayConfig.AckTimeout = 5s`.
- Per-connection push attempt timeout: `pushToConnection` waits up to `1s`.
- Cross-gateway forward request timeout: `2s` per attempt.
- Cross-gateway retry policy: `maxRetries=2` (total up to 3 attempts), `retryBackoff=50ms`.
- Circuit breaker: opens after 3 consecutive failures for 2s.

### Capacity Model (semi-quantitative)

#### Single-node gateway (no remote forward)

- **Ingress ceiling (rate-limit bound):**
  - Approx upper bound per active connection: `<= 100 msg/s` (before downstream checks).
- **Local fanout cost driver:**
  - CPU + memory copy + JSON marshal + channel enqueue per target connection.
  - With 2.1 dedup logic, duplicate group-member/user-device references no longer multiply local sends.

#### Dual/multi-gateway topology

- **Remote fanout cost driver (post-2.2):**
  - Forward calls scale by **unique remote gateway nodes**, not raw remote device count.
  - Worst-case remote latency budget per call chain is bounded by retry policy (`~3 * 2s + backoff`).
- **Failure isolation behavior:**
  - Circuit breaker suppresses repeated expensive attempts during outage windows.
  - Same-node multi-device failure now avoids duplicated failing forwards.

#### ACK lifecycle path

- **State cardinality driver:** in-flight messages awaiting ACK.
- **Timeout processing driver:** timer callbacks and status transitions.
- **Post-2.3 safety properties:**
  - stale timer no longer mutates newer pending state;
  - duplicate resolve does not inflate success metrics.

### Bottleneck Attribution (ranked)

1. **Network/remote dependency latency (highest volatility)**
   - Trigger: cross-gateway unavailable/timeout/resource exhaustion.
   - Impact: tail latency and transient delivery failures.
   - Evidence: retry/circuit-breaker tests and failure reason classification coverage.

2. **Serialization + per-target delivery overhead (high fanout scenarios)**
   - Trigger: large recipient fanout and repeated marshaling/delivery attempts.
   - Mitigation now present: gateway-node and device/member dedup reduce redundant work.

3. **ACK state contention and timer correctness (correctness-sensitive)**
   - Trigger: rapid register/resolve cycles under concurrency.
   - Mitigation now present: re-register stale timer isolation and resolve idempotency guards.

4. **Connection channel backpressure (localized saturation)**
   - Trigger: blocked/full send channels.
   - Behavior: non-blocking skip preserves progress for other recipients but may reduce delivered ratio.

### Operational Guidance Before Staging Load Baseline

- Prefer scaling by gateway-node partitioning to minimize per-request remote fanout width.
- Monitor forward failure reasons (`timeout/unavailable/transport_error`) as primary SLO leading signals.
- Watch ACK lifecycle counters (pending/timeout/late/success) to detect timer-pressure and delayed-delivery regressions.
- Keep circuit breaker thresholds conservative until 1.2/5.2 pressure evidence is available.

### Limitations

- This model does **not** provide authoritative throughput/P95/P99 values.
- Final numeric capacity envelope requires completion of 1.2/5.2 (staging-level pressure evidence).

## Phased Rollout & Rollback Strategy (Task 5.1)

### Preconditions (for this non-loadtest round)

Rollout can proceed only when all of the following are true:

1. Local quality gates pass (`lint/test/build`).
2. Cross-gateway integration subset passes.
3. Reliability regression suite (failure + recovery) passes.
4. OpenSpec non-loadtest tasks are complete for this wave (`2.1/2.2/2.3/3.1/3.2/3.3/4.1/4.2/4.3/1.3/2.4`).

> Note: `1.2` and `5.2` (staging pressure gates) are intentionally deferred by decision; production rollout should be conservative and reversible.

### Rollout Phases

#### Phase 0 — Canary (single gateway node)

- Scope: 1 node / minimal traffic slice.
- Duration: 30–60 minutes of steady traffic.
- Observe:
  - health endpoint (`/health`) availability;
  - cross-gateway forward failure ratio;
  - ACK timeout and late-ACK trend;
  - message delivery latency trend.

Promotion criteria:

- No sustained service-unhealthy condition.
- No sustained cross-gateway failure-rate spike versus pre-rollout baseline.
- ACK timeout trend stable (no persistent upward drift).

#### Phase 1 — Partial rollout (≈25% nodes)

- Scope: ~25% gateway nodes.
- Duration: 1–2 hours.
- Observe same indicators plus pod resource headroom.

Promotion criteria:

- Canary criteria still satisfied.
- No monotonic degradation in delivery success trend.
- Resource usage remains within planned envelope for current load.

#### Phase 2 — Broad rollout (≈50–75% nodes)

- Scope: staged expansion in 1–2 steps.
- Duration: 2–4 hours total.
- Continue monitoring + incident watch channel.

Promotion criteria:

- No P1/P0 incident triggered by this change set.
- Error/timeout behavior remains bounded and recoverable.

#### Phase 3 — Full rollout (100%)

- Scope: all gateway nodes.
- Post-rollout observation window: 24 hours.
- Keep rollback readiness (previous image/tag and deployment manifest) for the entire window.

### Rollback Triggers (threshold + duration)

Trigger immediate rollback when any condition persists beyond the specified duration:

1. **Service health instability**
   - Condition: repeated unhealthy responses on `/health`.
   - Duration: > 5 minutes.

2. **Cross-gateway forwarding degradation**
   - Condition: forward failure ratio materially above normal operating baseline.
   - Duration: > 5 minutes.

3. **ACK lifecycle degradation**
   - Condition: timeout/late-ACK metrics show persistent upward shift after rollout.
   - Duration: > 10 minutes.

4. **User-visible delivery regression**
   - Condition: sustained increase in delivery failures or severe latency complaints correlated with rollout window.
   - Duration: > 10 minutes.

### Rollback Procedure (operational)

1. Freeze further rollout progression.
2. Roll back deployment to last known-good version/tag.
3. Verify:
   - pods healthy and ready;
   - key service metrics return toward pre-rollout baseline;
   - no ongoing error amplification.
4. Keep rollback state for incident review and root-cause analysis.
